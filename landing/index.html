The problem with running agents on Linux

AI agents need autonomy. But today's infrastructure gives them either too much access or not enough.

🔓

No fine-grained isolation

A compromised agent in Docker can still probe the network, read env vars, and access shared volumes. Containers weren't designed for autonomous programs that take actions.

💬

No native communication

50 agents need Redis, RabbitMQ, or HTTP APIs just to talk to each other. That's 3 external services for basic message passing.

💥

No crash resilience

When an agent crashes, nothing restarts it with the right context. Kubernetes can restart pods, but it doesn't understand agent supervision trees.

⚠️

Security teams say no

Enterprises want to deploy agents but can't. "How do we let AI act autonomously without giving it the keys to the kingdom?" Nobody has a good answer.

Oxide OS: agents are the kernel's job

What if isolation, communication, and supervision were built into the operating system itself?

Capability Linux + Docker Oxide OS
Agent isolation Container namespaces (heavyweight, bypassable) Kernel capability tokens (unforgeable, zero overhead)
Network access control iptables rules (complex, misconfigurable) Per-agent, per-endpoint capability: net:api.openai.com:443
Agent communication External service (Redis, gRPC, HTTP) Built-in IPC: messages, shared memory, pub/sub
Crash recovery Kubernetes pod restart (no context) Supervision trees: RestartOne, RestartAll, Escalate
Permission delegation Not possible (root or nothing) Capability delegation with subset enforcement
Permission revocation Restart the container Instant cascading revocation (parent revokes all children)

Design principles

Every decision in Oxide OS follows these principles.

01

Zero ambient authority

An agent with no capabilities can do nothing. Every resource access — network, storage, IPC, spawning children — requires an explicit, unforgeable capability token.

cap #2 -> Agent(4) [WRITE] // Agent 2 can message Agent 4 // Cannot spawn, cannot network, // cannot read storage
02

Crash is expected, not exceptional

AI agents fail. Models hallucinate. APIs timeout. The kernel manages supervision trees with automatic restart policies — like Erlang, but for AI agents.

supervisor (RestartOne) ├─ researcher-1 [Running] ├─ researcher-2 [Crashed -> Restarting] └─ aggregator [Running]
03

Real hardware, not emulation

Oxide OS discovers PCI devices, sets up DMA ring buffers, and talks to virtio hardware directly. This is a real kernel, not a hosted runtime.

[pci] 00:03.0 vendor=1AF4 device=1000 [net] virtio-net: ONLINE (real driver) [net] Queue 0: 256 entries (RX) [net] Queue 1: 256 entries (TX)
04

Rust safety at every layer

Memory safety without runtime cost. The entire kernel is Rust with unsafe only at hardware boundaries. No garbage collector, no runtime, no C.

5,384 lines of Rust 0 lines of C 8 unsafe blocks (hardware I/O only)
05

Delegation, not escalation

An agent can give its children a subset of its own capabilities. Never more. Capabilities flow down the tree and can be revoked instantly, cascading to all descendants.

supervisor [SPAWN|KILL|DELEGATE] ├─ agent-1 [WRITE to agent-3] └─ agent-2 [READ from storage] // agent-2 CANNOT delegate // what it doesn't have
06

Built for the agent era

Not a general-purpose OS with agents bolted on. Every kernel primitive — scheduling, memory, IPC, security — is designed for autonomous AI agent workloads.

struct AgentConfig { system_prompt: String, model: ModelBinding, tools: Vec<String>, capabilities: Vec<CapId>, restart_policy: RestartPolicy, }
